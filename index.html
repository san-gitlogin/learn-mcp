Lesson 1 of 10 — Before We Start

⚡ How Does Claude Actually Work?

Before we build anything with MCP, let's make sure we're on the same page about how AI models like Claude work behind the scenes.

Claude is a Large Language Model (LLM) built by Anthropic. You send it a message (a "prompt"), and it sends back a response. That's it — it's a conversation over an API.

Every time you chat with Claude on claude.ai, your browser is making API calls behind the scenes. When we build our own apps, we make those same API calls ourselves using Python.

Calling Claude from Python

python 
from anthropic import Anthropic

# Create a client (uses ANTHROPIC_API_KEY env variable)
client = Anthropic()

# Send a message, get a response
message = client.messages.create(
    model="claude-sonnet-4-0",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "What is MCP?"}
    ]
)

print(message.content[0].text)
Anthropic()Creates a client. It automatically reads your ANTHROPIC_API_KEY from environment variables.
messages.create()Sends your conversation to Claude and waits for a response. This is the core API call.
model="claude-sonnet-4-0"Which Claude model to use. Sonnet is fast & affordable. Opus is the most powerful.
messages=[...]The conversation history. Each message has a role ("user" or "assistant") and content.

Pre-requisites

1. Python 3.10+ — Check with python --version

2. An Anthropic API Key — Get one at console.anthropic.com. Set it as an environment variable:
export ANTHROPIC_API_KEY="sk-ant-..."

3. uv (recommended) — A fast Python package manager. Install: pip install uv

4. The MCP SDK — pip install "mcp[cli]" anthropic

Not using Claude? Other options exist.

💡 MCP works with any LLM. While this guide uses Claude (Anthropic's API), the MCP protocol itself is model-agnostic. You could swap in:

• Ollama — Run open-source models locally (Llama, Mistral, etc.)
• Azure OpenAI — GPT-4 via Microsoft Azure
• OpenAI directly — GPT-4, GPT-4o
• Google Gemini — Via their API

The MCP server/client code stays the same. Only the "call the LLM" part changes.

Lesson 4 of 10

🏗️ Build Your First MCP Server

Building an MCP server in Python is embarrassingly simple. It's decorators on functions. If you can write a Python function, you can build an MCP server.

You're building a Notes Manager. Users want Claude to be able to read their notes and edit them.
1

Start the server (1 line)

Copy-paste boilerplate. Forget about it.

python
from mcp.server.fastmcp import FastMCP

# That's it. You have a server now.
mcp = FastMCP("NotesMCP")
2

Add some data

Just a Python dict. Nothing fancy.

python
notes = {
    "todo.md": "Buy groceries, call mom, fix the bug",
    "ideas.md": "App idea: dog walking Uber",
}
3

Make a Tool (let Claude read notes)

A decorator + a normal function. That's the whole pattern.

python
from pydantic import Field

@mcp.tool(
    name="read_note",
    description="Read a note by its filename",
)
def read_note(
    note_id: str = Field(description="Filename of the note"),
):
    if note_id not in notes:
        raise ValueError(f"Note {note_id} not found")
    return notes[note_id]
💡 "But what's Field?"
It's just a label for Claude. When Claude sees this tool, it reads the description to understand what note_id means. Without it, Claude would just see a parameter named note_id with no context. That's it. It's a hint for the AI.
4

Run it

One line at the bottom.

python
if __name__ == "__main__":
    mcp.run(transport="stdio")
🧠 The pattern to remember
Every MCP server tool follows this exact recipe:

1. @mcp.tool() — "Hey MCP, register this"
2. def my_function() — Just a normal Python function
3. Field(description=...) — Labels for Claude to understand the params
4. Return something or raise an error

That's it. There is no step 5.

Lesson 5 of 10

🔌 The Client: Your Phone Call to the Server

📱 Phone Call Analogy
Connecting = Dialing the number (starting the server process)
list_tools() = "Hey, what can you do?"
call_tool() = "Okay, do this specific thing for me"
cleanup() = Hanging up the phone

Here are the only 5 methods you ever need on a client:

python
# 1. "What tools do you have?"
tools = await client.list_tools()

# 2. "Run this tool with these inputs"
result = await client.call_tool("read_note", {"note_id": "todo.md"})

# 3. "Fetch this data for me"
data = await client.read_resource("notes://all")

# 4. "What prompts do you have?"
prompts = await client.list_prompts()

# 5. "Give me this prompt filled in with these args"
messages = await client.get_prompt("format", {"note_id": "todo.md"})
💡 You don't need to memorize the client code. The connection boilerplate is always the same — you write it once and copy it forever. What matters is knowing these 5 methods exist and when to call them.

Here's the magic. Your app does this in a loop:

1. Get tools from server → send them to Claude with user's question
2. Claude says "I want to call read_note" → you call it via client
3. Send tool result back to Claude → Claude says "I want to call edit_note"
4. You call it → send result back → Claude gives final answer
5. Loop until Claude stops asking for tools

This loop is the heartbeat of every MCP application.

Lesson 6 of 10

📂 Resources & Prompts in 2 Minutes

You already know tools. Resources and prompts use the exact same decorator pattern — just a different decorator name.

📂 Resource (your app fetches data)

@mcp.resource("notes://all")
def list_all_notes():
    return list(notes.keys())

@mcp.resource("notes://{id}")
def get_note(id: str):
    return notes[id]

✨ Prompt (user triggers a workflow)

@mcp.prompt(name="format")
def format_note(
    id: str = Field(...)
):
    return [
        base.UserMessage(
            f"Reformat {id}..."
        )
    ]
🧠 The @ mentions trick
Here's a slick pattern: when a user types @todo.md in their message, your app code:

1. Detects the @ mention
2. Calls read_resource("notes://todo.md") — fetches the note content from the server
3. Injects the content directly into the prompt sent to Claude
4. Claude sees the note content immediately — no tool call needed!

This is way faster than Claude having to call a tool to read the note. The resource was fetched by your code, not by Claude. Your app already had the data before Claude even saw the message.
💡 Resource URIs work like URLs:
notes://all — static, always returns the same type of thing (list of all note filenames)
notes://{id} — dynamic, the {id} part becomes a function parameter automatically

Lesson 7 of 10

🧠 Sampling: "Hey Client, Call Claude For Me"

This is the most mind-bending feature. Ready?

Imagine you've built a public MCP server for research — think of it like a free plugin anyone can install. It has a tool that scrapes Wikipedia, gathers articles, and compiles data on any topic. (Example: a user connects to your server and asks "research quantum computing".)

Now, after gathering all that raw data, the tool needs to summarize it into a nice report. Summarization needs an LLM like Claude.

The problem: Since users connect to your server, every request flows through your code. If you put your own Anthropic API key on the server to call Claude for summarization, then every user's request costs you money. Your server goes viral — congrats, you now owe Anthropic $50,000. Not great.
🧠 The "Reverse Uno" analogy
Normal flow: Client asks Server to do stuff.
Sampling: Server asks Client "hey, YOU call Claude for me and send me the result."

It's like a restaurant (server) saying to the customer (client): "We prepared your ingredients, but can you cook the final dish at your house? You've got a better kitchen — and your own gas bill."

The client has the API key. The client pays. The server stays cheap to run.

Server Side — Asking the client to call Claude

python — server
@mcp.tool()
async def summarize(text: str, ctx: Context):
    result = await ctx.session.create_message(
        messages=[SamplingMessage(
            role="user",
            content=TextContent(type="text", text=text)
        )],
        max_tokens=4000,
    )
    return result.content.text
🔁 Remember: @mcp.tool() registers this as a tool — something Claude (or a client) can call. The new piece here is ctx: Context. This special object gives the tool a way to talk back to the client. ctx.session.create_message() sends a request backwards from server to client, saying "please generate text for me."
@mcp.tool()Registers summarize as a tool on the server. A connected client (and Claude) can call it.
ctx: ContextThe server's "walkie-talkie" back to the client. It lets the tool send requests back to whoever connected.
ctx.session.create_message()The sampling call. Says: "Client, here's a prompt. Please send it to an LLM and give me back the response."
SamplingMessage(...)An MCP-formatted message. The client will convert this into whatever format its LLM needs.
result.content.textThe LLM's generated text comes back here. The server can use it, process it, or return it.

Client Side — Handling the server's request

python — client
async def sampling_callback(context, params):
    text = await chat(params.messages)  # YOUR function that calls Anthropic API
    return CreateMessageResult(
        role="assistant", model=model,
        content=TextContent(type="text", text=text),
    )

# Pass it when connecting to the server
ClientSession(read, write, sampling_callback=sampling_callback)
sampling_callbackA function YOU write. It gets called automatically when the server sends a create_message request.
params.messagesThe messages the server wants you to send to Claude. These are in MCP format — you may need to convert them.
chat(params.messages)YOUR function that calls the Anthropic API. This is where the API key and cost lives — on the client side.
sampling_callback=...You pass your callback when creating the connection. Now the server can request text generation at any time.

The Full Communication Flow

1
Client calls session.call_tool("summarize", {"text": "..."})
2
Server receives the call, starts running the summarize tool function
3
Server hits ctx.session.create_message() — sends a request BACK to the client: "Please call an LLM with this prompt"
4
Client's sampling_callback fires automatically. It calls the Anthropic API using the CLIENT's API key
5
Claude API returns generated text to the client
6
Client wraps the text in CreateMessageResult and sends it back to the server
7
Server receives the text in result.content.text and returns it as the tool's output

Lesson 8 of 10

🔐 Roots & Notifications

Roots = "You can look here, but not there"

🧠 Hotel Keycard Analogy
Roots are like hotel keycards. Your keycard opens your room and the gym, but not other guest rooms. You give the MCP server a list of folders it's allowed to access. It can't snoop anywhere else.

Bonus: Claude can ask "what folders do I have access to?" (list_roots) and then "what's in that folder?" (read_dir) — instead of the user typing full file paths.

Step 1: The Client Defines Which Folders Are Allowed

The client is the one who decides which folders the server can access. Typically, the user provides these paths (e.g., as command-line arguments when launching the app):

python — client: defining roots
from pathlib import Path
from pydantic import FileUrl
from mcp.types import Root, ListRootsResult

# User says: "Server, you can access these two folders"
root_paths = ["/Users/me/Movies", "/Users/me/Desktop"]

# Convert them to Root objects (MCP's format)
roots = []
for path in root_paths:
    p = Path(path).resolve()
    file_url = FileUrl(f"file://{p}")
    roots.append(Root(uri=file_url, name=p.name))

# Create a callback — when the server asks "what roots do I have?",
# this function responds with the list
async def handle_list_roots(context):
    return ListRootsResult(roots=roots)

# Pass the callback when creating the session
ClientSession(read, write, list_roots_callback=handle_list_roots)
root_paths = [...]These are the actual folder paths. Usually the user provides them (e.g., uv run main.py /Users/me/Movies).
Root(uri=..., name=...)From MCP. A Root object wraps a folder path in a file:// URI. The MCP spec requires all roots to use this format.
handle_list_roots()Your callback. The server doesn't get the roots immediately. Instead, it can ask for them later, and this callback responds.
list_roots_callback=...You wire the callback into the ClientSession. Now whenever the server calls ctx.session.list_roots(), this fires.
💡 The flow: User provides folder paths → Client converts them to Root objects → Client registers a callback → Server calls ctx.session.list_roots() whenever it needs them → Callback fires and sends the list back. The server never gets the roots upfront — it asks for them on demand.

Step 2: The Server Checks Roots Before Accessing Files

Now on the server side, every tool that touches the file system should verify the path is allowed:

python — server: using roots in a tool
@mcp.tool()
async def convert_video(input_path: str, format: str, *, ctx: Context):
    input_file = Path(input_path)

    # ⛔ CHECK BEFORE DOING ANYTHING
    if not await is_path_allowed(input_file, ctx):
        raise ValueError("Access denied!")

    return await VideoConverter.convert(input_path, format)
⚠️ Where does is_path_allowed come from?
It is NOT part of the MCP library. You write it yourself. The MCP SDK gives you ctx.session.list_roots() (which triggers the client's callback you saw in Step 1 above) — but checking the requested path against that list is YOUR responsibility. Here's what that function looks like:
python — you write this yourself
async def is_path_allowed(requested_path: Path, ctx: Context) -> bool:
    # Ask the client: "What roots (folders) did you allow?"
    roots_result = await ctx.session.list_roots()  # ← this IS from MCP
    client_roots = roots_result.roots

    if not requested_path.exists():
        return False

    # Check: is the requested path inside any allowed root?
    for root in client_roots:
        root_path = file_url_to_path(root.uri)
        try:
            requested_path.relative_to(root_path)  # subfolder check
            return True
        except ValueError:
            continue

    return False
ctx.session.list_roots()From MCP. Asks the client "what folders did you allow?" Client responds with a list of Root objects.
relative_to(root_path)Standard Python. Checks if the requested path is inside the allowed folder. Throws ValueError if not.
is_path_allowed()Your own function. MCP does NOT enforce roots automatically — you must call this in every tool that touches files.

Step 3: Give Claude Tools to Discover Files

Remember the hotel keycard analogy? is_path_allowed is the lock on the door. But Claude also needs a way to find which doors exist. That's where list_roots and read_dir come in — two tools you define on the server so Claude can explore the allowed folders:

python — server: let Claude discover files
@mcp.tool()
async def list_roots(ctx: Context):
    """List all directories that are accessible to this server."""
    roots_result = await ctx.session.list_roots()
    client_roots = roots_result.roots
    return [file_url_to_path(root.uri) for root in client_roots]


@mcp.tool()
async def read_dir(
    path: str = Field(description="Path to a directory to read"),
    *,
    ctx: Context,
):
    """Read directory contents. Path must be within an allowed root."""
    requested_path = Path(path).resolve()

    if not await is_path_allowed(requested_path, ctx):
        raise ValueError("Can only read directories within a root")

    return [entry.name for entry in requested_path.iterdir()]
list_rootsWhen Claude asks "what folders can I see?", this tool calls ctx.session.list_roots() (which triggers the client's callback from Step 1) and returns human-readable paths.
read_dirWhen Claude asks "what's inside /Users/me/Movies?", this tool lists the folder contents — but only if the path passes the is_path_allowed check first.
*, ctx: ContextThe * before ctx means it's a keyword-only argument. MCP injects Context automatically — the user never provides it.
🧠 How Claude uses all three together
User says: "Convert biking.mp4 to MOV"

1. Claude calls list_roots() → gets back ["/Users/me/Movies", "/Users/me/Desktop"]
2. Claude calls read_dir("/Users/me/Movies") → gets back ["biking.mp4", "vacation.mp4"]
3. Claude calls convert_video("/Users/me/Movies/biking.mp4", "mov") → which checks is_path_allowed → ✅ allowed → conversion runs

The user just said "biking.mp4" — Claude figured out the full path on its own by exploring the roots. No full paths required from the user.

Notifications = "Hey user, I'm still working..."

Long-running tools can feel broken without feedback. Two server-side calls fix this:

python — server
@mcp.tool()
async def research(topic: str, ctx: Context):
    await ctx.info("Searching Wikipedia...")      # Log message
    await ctx.report_progress(30, 100)             # 30% done

    data = await scrape(topic)

    await ctx.info("Writing report...")
    await ctx.report_progress(80, 100)             # 80% done
    return summarize(data)
💡 But how does the client actually SEE these? The server emits notifications, but the client needs callbacks to catch them — just like catching a thrown ball:
1
Server calls ctx.info("Searching...") inside a tool
2
MCP protocol automatically sends a LoggingMessageNotification JSON message over the wire to the client
3
Client's logging_callback fires automatically — prints "Searching..." to the terminal (or shows it in a UI)
python — client: catching notifications
# What happens when the server sends a log message
async def logging_callback(params):
    print(params.data)  # prints "Searching Wikipedia..."

# What happens when the server reports progress
async def progress_callback(progress, total, message):
    percentage = (progress / total) * 100
    print(f"Progress: {percentage:.0f}%")  # prints "Progress: 30%"

# Wire them up:
# logging_callback → goes into ClientSession (catches ALL log messages)
ClientSession(read, write, logging_callback=logging_callback)

# progress_callback → goes into a specific call_tool (catches progress for THIS call)
session.call_tool("research", {...}, progress_callback=progress_callback)

🚀 Transport: How Messages Travel

MCP messages are just JSON. But how they get from A to B depends on the transport.

📟 Stdio (Development)

Client and server on the same machine. Talk via stdin/stdout. Like two people in the same room passing notes.

mcp.run(transport="stdio")

✅ Full features
✅ Simple setup
❌ Same machine only

🌐 Streamable HTTP (Production)

Client and server on different machines. Talk over HTTP. Like a phone call across the world.

mcp.run(transport="streamable-http")

✅ Remote servers
✅ Scalable
⚠️ Some features may be limited

⚠️ The Two Scary Flags

When using HTTP transport, two config flags can dramatically change what your server can do. Here's exactly where they go in your code:

python — where the flags go
from mcp.server.fastmcp import FastMCP

# These flags are set when you CREATE the server
mcp = FastMCP(
    "my-server",
    stateless_http=True,   # ⚠️ Enables scaling, BUT kills sampling + notifications
    json_response=True,    # ⚠️ Simpler responses, BUT kills streaming progress
)

# Then run with HTTP transport
mcp.run(transport="streamable-http")
stateless_http=TrueServer forgets clients between requests. Good for load balancers (multiple server instances), but kills: sampling, progress, roots, subscriptions.
json_response=TrueResponses come as plain JSON instead of SSE streams. Kills: real-time progress updates and log messages during tool execution.
Both False (default)All features work — sampling, notifications, progress, roots. Use this unless you specifically need horizontal scaling.
💡 Rule of thumb: Start with both False (the default). Only set to True when you need horizontal scaling and understand the tradeoffs. If your server uses sampling or progress notifications, you cannot use stateless_http=True.
